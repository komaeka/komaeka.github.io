Hyperledger Fabirc学习笔记

发表于2023-07-09|计算机学习心得

安装

启动网络

部署智能合约（链码）

设置监控智能合约的日志（可选操作）

  1. 启动网络并创建通道：进入test-network目录，运行./network.sh up createChannel
  2. 运行./monitordocker.sh，此时会额外启动一个名为logspout的容器

准备工作

  • 命令执行目录默认为fabric-samples
  • 要想使用peer CLI，需要三个前提：
    • 网络已经启动
    • 设置FABRIC_CFG_PATH环境变量，指向fabric-samples中的core.yaml文件：export FABRIC_CFG_PATH=$PWD/config/
    • 设置peer节点的环境变量，这将决定以谁的身份操作peer CLI

打包智能合约

针对Go,Java,JavaScript,Typescript不同语言所编写的智能合约，步骤会有所不同，此处仅记录使用Go语言编写的智能合约的打包步骤

  1. 进入fabric-samples/asset-transfer-basic/chaincode-go，运行GO111MODULE=on go mod vendor
  2. 使用peer CLI创建链码包：peer lifecycle chaincode package basic.tar.gz --path asset-transfer-basic/chaincode-go/ --lang golang --label basic_1.0

安装链码包

由于将背书策略设置为要求Org1和Org2都背书，所以需要在两个组织peer节点上安装链码

设置环境变量，分别以Org1/Org2管理员用户身份操作peer CLI安装链码包：peer lifecycle chaincode install basic.tar.gz

批准链码定义

安装链码包后，需要批准组织的链码定义。该定义包括链码治理的重要参数，例如名称、版本和链码背书策略

  1. 使用peer CLI查询PackageID（PackageID是链码标签和链码二进制文件的哈希值的组合。每个peer节点都会生成相同的包PackageID）：peer lifecycle chaincode queryinstalled
  2. 将PackageID保存为环境变量：export CC_PACKAGE_ID=basic_1.0:e4de097efb5be42d96aebc4bde18eea848aad0f5453453ba2aad97f2e41e0d57
  3. 设置环境变量，分别以Org1/Org2管理员用户身份操作peer CLI批准链码定义：peer lifecycle chaincode approveformyorg -o localhost:7050 --ordererTLSHostnameOverride orderer.example.com --channelID mychannel --name basic --version 1.0 --package-id $CC_PACKAGE_ID --sequence 1 --tls --cafile "${PWD}/test-network/organizations/ordererOrganizations/example.com/orderers/orderer.example.com/msp/tlscacerts/tlsca.example.com-cert.pem"

将链码定义提交到通道

当足够数量的组织批准链码定义后，一个组织可以将链码定义提交到通道。如果大多数通道成员批准了该定义，则提交交易将成功，并且链码定义中商定的参数将在通道上实现

  1. 使用peer CLI检查通道成员是否批准了相同的链码定义，该命令将生成一个JSON映射，显示频道成员是否批准了checkcommitreadiness命令中指定的参数：peer lifecycle chaincode checkcommitreadiness --channelID mychannel --name basic --version 1.0 --sequence 1 --tls --cafile "${PWD}/test-network/organizations/ordererOrganizations/example.com/orderers/orderer.example.com/msp/tlscacerts/tlsca.example.com-cert.pem" --output json
  2. 由组织管理员使用peer CLI将链码定义提交到通道：peer lifecycle chaincode commit -o localhost:7050 --ordererTLSHostnameOverride orderer.example.com --channelID mychannel --name basic --version 1.0 --sequence 1 --tls --cafile "${PWD}/test-network/organizations/ordererOrganizations/example.com/orderers/orderer.example.com/msp/tlscacerts/tlsca.example.com-cert.pem" --peerAddresses localhost:7051 --tlsRootCertFiles "${PWD}/test-network/organizations/peerOrganizations/org1.example.com/peers/peer0.org1.example.com/tls/ca.crt" --peerAddresses localhost:9051 --tlsRootCertFiles "${PWD}/test-network/organizations/peerOrganizations/org2.example.com/peers/peer0.org2.example.com/tls/ca.crt"
  3. 使用peer CLI确认链码定义是否已提交到通道，如果链码成功提交到通道，将返回链码定义的序列和版本：peer lifecycle chaincode querycommitted --channelID mychannel --name basic

调用链码

使用peer CLI调用链码

  • 在账本上创建一组初始资产：peer chaincode invoke -o localhost:7050 --ordererTLSHostnameOverride orderer.example.com --tls --cafile "${PWD}/test-network/organizations/ordererOrganizations/example.com/orderers/orderer.example.com/msp/tlscacerts/tlsca.example.com-cert.pem" -C mychannel -n basic --peerAddresses localhost:7051 --tlsRootCertFiles "${PWD}/test-network/organizations/peerOrganizations/org1.example.com/peers/peer0.org1.example.com/tls/ca.crt" --peerAddresses localhost:9051 --tlsRootCertFiles "${PWD}/test-network/organizations/peerOrganizations/org2.example.com/peers/peer0.org2.example.com/tls/ca.crt" -c '{"function":"InitLedger","Args":[]}'
  • 使用查询函数来读取由链码创建的汽车集：peer chaincode query -C mychannel -n basic -c '{"Args":["GetAllAssets"]}'

使用Fabric Gateway客户端API构建的示例程序来调用链码

简介

在此示例中，将使用TypeScript智能合约

  • 资产转移示例演示了如何创建、更新和查询资产。它涉及以下两个组成部分：
    • Sample application：调用区块链网络中智能合约的应用程序，位于fabric-samples/asset-transfer-basic/application-gateway-(go/java/typescript)目录中
    • Smart contract：智能合约，位于fabric-samples/asset-transfer-basic/chaincode-(go/java/javascript/typescript)目录中

一些猜想

  • 在面对填写peer节点的tlsRootCertFiles，即环境变量CORE_PEER_TLS_ROOTCERT_FILE的时候，fabric-samples中的shell脚本会使用organizations/peerOrganizations/org1.example.com/tlsca/tlsca.org1.example.com-cert.pem，而官方文档中则使用organizations/peerOrganizations/org1.example.com/peers/peer0.org1.example.com/tls/ca.crt。第一个是组织级别的，第二个则是针对组织下属的节点级别的，在两个都能用的情况下，我有理由怀疑用组织级别的TLS证书能够概括该组织下属的所有节点
